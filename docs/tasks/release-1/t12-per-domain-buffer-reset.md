# T12: Console/network buffers reset on same-tab domain change

## Goal

Each tab's console and network buffers must belong to the tab's current domain: when a tab navigates to a different hostname, the buffer is replaced with a fresh empty one, so a read never returns one site's traffic while the tab is on another site. Today both buffers are keyed by tabId only and survive cross-domain navigations, which contradicts the tool schemas' own promises ("Returns console messages from the current domain only"; "Requests are automatically cleared when the page navigates to a different domain"). Same-domain navigations, including SPA history-API URL changes, must keep the buffer.

## Project context

    MCP Client <--stdio--> Binary <--native messaging--> Extension <--CDP--> Browser

The two binary roles run as separate OS processes bridged by tokio-native named-pipe/UDS IPC. The binary relays tool calls to the extension and relays the extension's results back to the MCP client unchanged.

## Current behavior

All line numbers verified against `extension/service-worker.js` as of this writing.

- Line 14: `const consoleBuffer = new Map(); // tabId -> [{ level, text }]`
- Line 15: `const networkBuffer = new Map(); // tabId -> [{ requestId, method, url, status, mimeType }]`
- `ensureAttached(tabId)` (lines 55-64, with the `attaching` in-flight map at line 54) attaches the debugger and records `attached.set(tabId, { domains: new Set() })`. It does not record the tab's URL or hostname anywhere.
- The `chrome.tabs.onRemoved` listener (lines 125-133) detaches the debugger and deletes `consoleBuffer` (line 130), `networkBuffer` (line 131), and `screenshotCtx` (line 132) for the removed tab. This is the ONLY place buffers are deleted.
- The `chrome.debugger.onEvent` listener (lines 137-154) appends:
  - `Runtime.consoleAPICalled` -> `pushCapped(consoleBuffer, tabId, { level, text })` (line 145),
  - `Network.requestWillBeSent` -> `pushCapped(networkBuffer, tabId, { requestId, method, url, status: 0 })` (line 147),
  - `Network.responseReceived` -> reads `networkBuffer.get(tabId) || []` (line 149), finds the entry by `requestId`, and mutates its `status`/`mimeType` in place (lines 150-151); if not found it appends a response-only entry with `method: "?"` via `pushCapped` (line 152).
- `pushCapped(map, tabId, item)` (lines 155-160) appends to the tab's array and caps it at 1000 entries with `splice`.
- `read_console_messages(a)` (lines 512-526): group check, `ensureAttached`, `enableDomain(a.tabId, "Runtime")`, reads `consoleBuffer.get(a.tabId) || []` (line 517), applies `onlyErrors` and `pattern` filters and the limit slice, then `if (a.clear) consoleBuffer.set(a.tabId, []);` (line 524) and returns the joined `[level] text` lines or the no-match string (line 525).
- `read_network_requests(a)` (lines 527-536): same shape with `enableDomain(a.tabId, "Network")`, reads `networkBuffer.get(a.tabId) || []` (line 531), applies the `urlPattern` substring filter and the limit slice, then `if (a.clear) networkBuffer.set(a.tabId, []);` (line 534) and returns the joined request lines or the no-match string (line 535).
- There is no persistent `chrome.tabs.onUpdated` listener. The only `tabs.onUpdated` use is the transient per-navigation listener inside `waitForLoad` (lines 343-354), which is removed once the load completes or after 10 seconds. Do not touch it.
- CDP domains are enabled lazily by `enableDomain` (lines 118-124): `Runtime` and `Network` events only start flowing after the first read call on a tab.

The defect: nothing keys a buffer to a domain. Navigate a tab from a.com to b.com and a subsequent `read_console_messages` or `read_network_requests` happily returns a.com's messages and requests, leaking one site's traffic into another site's results and contradicting the schema descriptions quoted above.

## Required behavior

Give each buffer an owning hostname and reset it whenever the tab's hostname no longer matches. Implement exactly the following in `extension/service-worker.js`.

1. Hostname helper. Add a small function:

       function hostOf(url) {
         try { return new URL(url).hostname; } catch { return ""; }
       }

   Semantics: the owning domain is the FULL hostname of the tab URL, compared with exact string equality. Subdomains are distinct (`app.example.com` is not `example.com`). URLs without a hostname (`about:blank`, `file://...`, unparsable strings) yield `""`, which is a legitimate owner value; navigating from `example.com` to `about:blank` is a domain change and resets the buffer. No eTLD+1 or public-suffix logic.

2. Hostname tracking. Add module-level state next to the buffer declarations:

       const tabHost = new Map(); // tabId -> hostname of the tab's current URL ("" when none)

   Keep it current three ways:

   a. A persistent `chrome.tabs.onUpdated` listener registered at module scope (place it in the console/network buffering section near the `chrome.debugger.onEvent` listener):

          chrome.tabs.onUpdated.addListener((tabId, info) => {
            if (info.url !== undefined) tabHost.set(tabId, hostOf(info.url));
          });

      This fires for full navigations and for SPA history-API URL changes. A same-hostname URL change stores the same value and therefore triggers no reset.

   b. Seed on attach: inside the attach path of `ensureAttached` (the async closure at lines 58-61), immediately after `attached.set(tabId, { domains: new Set() })`, resolve the tab and record its hostname:

          try {
            const t = await chrome.tabs.get(tabId);
            tabHost.set(tabId, hostOf(t.url || ""));
          } catch { /* tab gone */ }

      A failed `chrome.tabs.get` must not fail the attach.

   c. Refresh on read: see step 6.

   Also extend the `chrome.tabs.onRemoved` listener to `tabHost.delete(tabId);` alongside the existing buffer deletions at lines 130-132.

3. Buffer shape. Change both maps from `tabId -> array` to `tabId -> { host, items }`, where `host` is the owning hostname recorded when the buffer was created (it may be `undefined` when the hostname was not yet known) and `items` is the array of entries in exactly today's per-entry shapes (`{ level, text }` for console; `{ requestId, method, url, status, mimeType }` for network). Update the declaration comments on lines 14-15 to describe the new shape.

4. Ownership rule. Add one helper that both the append path and the read path use:

       function bufferFor(map, tabId, host) {
         let buf = map.get(tabId);
         if (!buf || (host !== undefined && buf.host !== undefined && buf.host !== host)) {
           buf = { host, items: [] };
           map.set(tabId, buf);
         } else if (buf.host === undefined && host !== undefined) {
           buf.host = host; // entries captured before the host was known belong to the first host learned
         }
         return buf;
       }

   In words: reset only when both the buffer's owner and the current hostname are known and differ. When the buffer's owner is unknown (created before the hostname was learned, for example right after a service-worker restart) and the hostname becomes known, adopt it without discarding the entries. When the current hostname is unknown, keep the buffer as is.

5. Append path. Rewrite `pushCapped` to route through `bufferFor`:

       function pushCapped(map, tabId, item) {
         const buf = bufferFor(map, tabId, tabHost.get(tabId));
         buf.items.push(item);
         if (buf.items.length > 1000) buf.items.splice(0, buf.items.length - 1000);
       }

   The cap stays at 1000. Update the `Network.responseReceived` join (lines 148-152) for the new shape: obtain `const buf = bufferFor(networkBuffer, tabId, tabHost.get(tabId));`, then `find` by `requestId` inside `buf.items` and mutate `status`/`mimeType` in place as today; when not found, fall back to the existing response-only `pushCapped` append. Note the consequence: if the domain changed between request and response, the fresh buffer will not contain the request entry, so the response lands as a response-only entry in the new domain's buffer. That is correct.

6. Read path. In BOTH `read_console_messages` and `read_network_requests`, after the existing `enableDomain` call, resolve the tab's current hostname fresh and apply the ownership rule before filtering:

       const tab = await chrome.tabs.get(a.tabId);
       const host = hostOf(tab.url || "");
       tabHost.set(a.tabId, host);
       const buf = bufferFor(consoleBuffer, a.tabId, host); // networkBuffer in the other handler
       let msgs = buf.items; // let reqs = buf.items;

   This guarantees the no-leak property even when no event has been appended since a cross-domain navigation: the stale buffer is replaced with a fresh empty one owned by the new hostname and the read returns the existing no-match string. All filter, limit, and output logic stays exactly as it is (the filters and the final `slice` produce copies, so the returned data never aliases `buf.items` mutation).

   Update the `clear` lines for the new shape:

       if (a.clear) consoleBuffer.set(a.tabId, { host, items: [] });
       if (a.clear) networkBuffer.set(a.tabId, { host, items: [] });

7. Result text is unchanged. Every string the two handlers return, for both the entries case and the zero-entries case, must remain byte-identical to the current code. This task changes which entries are in the buffer, nothing about how they are printed. A buffer reset on domain change is documented contract behavior (it is in the schema descriptions), not a failure, so it needs no extra notice in the result text.

8. Accepted limitation (do not "fix" it). CDP's `Network.requestWillBeSent` for a cross-domain navigation's main document can arrive before `chrome.tabs.onUpdated` reports the new URL. That single entry is appended to the old domain's buffer and discarded when the reset fires on the next append or read. Its `Network.responseReceived` then appends a response-only entry (`method: "?"`) to the new buffer via the existing fallback. Accept this; do not add heuristics such as inspecting request URLs, checking `params.type === "Document"`, enabling the `Page` CDP domain for `Page.frameNavigated`, or requesting the `webNavigation` permission.

9. Comment style. One short comment on `bufferFor` (or the buffering section header) noting that buffers are owned by the tab's current hostname per the schema contract is appropriate. No other new comments.
